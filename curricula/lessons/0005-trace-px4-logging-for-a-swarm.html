Lesson 0005 · PX4 swarm logging roadmap

Trace one sample into a fleet log

Tangible win: after this lesson you can name the correct extension point for a swarm datum, a fleet transport, and cross-vehicle time alignment.

1. Start with the mental model

A PX4 module publishes a typed uORB message. The logger subscribes to configured topics, writes ULog schema records, and then frames every updated sample as a ULog data record. LogWriter fans those bytes to the file and MAVLink backends. The MAVLink module packetizes them as LOGGING_DATA or LOGGING_DATA_ACKED.

That means a swarm-specific measurement usually belongs in a uORB message. A swarm-wide collector belongs offboard. A new logger backend is the most invasive option, not the default.

2. See the serialization loop

if (copy_if_updated(sub_idx,
        _msg_buffer + sizeof(ulog_message_data_s), try_to_subscribe)) {
    const size_t msg_size = sizeof(ulog_message_data_s)
                          + sub.get_topic()->o_size_no_padding;
    _msg_buffer[2] = static_cast<uint8_t>(ULogMessageType::DATA);
    // bytes 3..4 contain the logged-message id
    write_message(LogType::Full, _msg_buffer, msg_size);
}

The original is in logger.cpp. ULog also embeds topic formats and firmware/vehicle metadata, so the receiver can parse a standalone file without a matching firmware checkout.

3. Select data without editing logger code

# /fs/microsd/etc/logging/logger_topics.txt
swarm_log 100       # all instances, at most 10 Hz
timesync_status 1000
vehicle_gps_position 200

The parser accepts <topic> [interval_ms [instance]]. For a prototype, PX4's debug_* messages already exist; for a durable plugin, add a typed SwarmLog.msg and list it here or add a dedicated logging-profile bit.

4. Reuse the MAVLink stream

The included Python client sends MAV_CMD_LOGGING_START, receives chunks, acknowledges reliable chunks, tracks 16-bit sequences, and reconstructs message boundaries after loss. A swarm collector generalizes its state from one object to a dictionary keyed by inbound MAVLink source system/component.

key = (m.get_srcSystem(), m.get_srcComponent())
state = streams[key]

if m.get_type() == "LOGGING_DATA_ACKED":
    link.mav.logging_ack_send(key[0], key[1], m.sequence)

is_new, drops = state.sequence.observe(m.sequence)
if is_new:
    state.ulog.feed(m.data[:m.length], m.first_message_offset, drops)

Keep a separate .ulg per vehicle. This preserves compatibility with PyULog and Flight Review; a small fleet manifest can map SYSID and sys_uuid to filenames and clock transforms.

5. Align clocks explicitly

ULog sample timestamps are local microseconds since boot. For GNSS-equipped vehicles, use vehicle_gps_position.time_utc_usec and its corresponding local sample timestamp to estimate a per-vehicle local-to-UTC transform. For companion-clock alignment, log timesync_status.estimated_offset and round-trip time. Do not use collector arrival timestamps as if they were sample time.

6. Retrieve before moving on

  1. Where should a formation error be represented: text log, custom uORB topic, or custom ULog bytes?
  2. Which identifier separates two vehicles' incoming stream state?
  3. Why can two equal PX4 timestamps still represent different real-world instants?

Expected answers: typed uORB topic; inbound source SYSID/component (plus persistent UUID in the file); each vehicle has an independent boot clock.
